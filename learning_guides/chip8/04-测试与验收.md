# 04：CHIP-8 测试与验收

本篇配合仓库中的测试套件：

```text
01_chip8_tests/
```

测试的目的不是只看“能不能打开游戏”，而是明确知道哪一类指令已经正确、哪一类仍有错误。

## 一、为什么必须使用测试 ROM

模拟器常见问题包括：

- `PC` 多加或少加 2；
- 指令字段 X/Y 提取错误；
- 进位、借位标志错误；
- `VF` 被过早覆盖；
- `FX55/FX65` 对 `I` 的行为不一致；
- Shift 指令兼容模式不同；
- 绘制边界行为不同；
- 键盘按下与释放状态错误；
- 定时器速度受 CPU 频率影响。

普通游戏可能暂时看起来能运行，但内部已经执行错误。测试 ROM 可以把错误缩小到某组 opcode。

## 二、测试前准备

你的模拟器至少支持命令行传入 ROM：

```text
chip8.exe path/to/test.ch8
```

建议增加调试参数：

```text
chip8.exe --trace path/to/test.ch8
chip8.exe --break 0x234 path/to/test.ch8
chip8.exe --cycles 100 path/to/test.ch8
```

至少准备以下日志：

```text
PC=0200 OPCODE=00E0 I=0000 SP=0
V0=00 V1=00 V2=00 V3=00 ... VF=00
```

## 三、测试顺序

## Test 1：CHIP-8 Logo

测试 ROM：

```text
01_chip8_tests/bin/1-chip8-logo.ch8
```

依赖指令：

```text
00E0  清屏
6XNN  给寄存器赋立即数
ANNN  设置 I
DXYN  绘制精灵
1NNN  跳转
```

预期：

- 大约执行 39 个周期后显示 CHIP-8 Logo；
- 继续执行后进入循环；
- 屏幕不会随机闪烁；
- PC 不会跑出 ROM 区域。

失败排查：

| 现象 | 优先检查 |
|---|---|
| 完全黑屏 | ROM 加载、PC 起始地址、`00E0`、`DXYN` |
| 图形位置错 | X/Y 字段、坐标计算 |
| 图形缺行 | N 字段、`I + row` |
| 图形乱码 | 指令大小端、精灵位顺序 |
| 很快崩溃 | `PC += 2`、跳转地址 |

验收：

- [ ] Logo 清晰可辨；
- [ ] 39 周期附近完成绘制；
- [ ] 跳转循环稳定。

## Test 2：IBM Logo

测试 ROM：

```text
01_chip8_tests/bin/2-ibm-logo.ch8
```

新增重点：

```text
7XNN  VX += NN
DXYN  非对齐位置绘制
```

预期：

- 大约 20 个周期后显示 IBM Logo；
- 非字节对齐的绘制仍然正确。

失败排查：

| 现象 | 优先检查 |
|---|---|
| Logo 倾斜或分裂 | 每个像素坐标计算 |
| 只有部分图案 | `7XNN` 或 `DXYN` 行循环 |
| 重画后不消失 | XOR 实现错误 |
| VF 永远为 0 | 碰撞检测顺序错误 |

验收：

- [ ] IBM Logo 正确；
- [ ] 非对齐绘制正确；
- [ ] XOR 和碰撞标志正确。

## Test 3：Corax+ Opcode Test

测试 ROM：

```text
01_chip8_tests/bin/3-corax+.ch8
```

主要覆盖：

```text
3XNN  4XNN  5XY0  9XY0
1NNN  2NNN  00EE
8XY0  8XY1  8XY2  8XY3
8XY4  8XY5  8XY6  8XY7  8XYE
FX1E  FX33  FX55  FX65
```

预期：

- 屏幕上对应指令位置显示通过标记；
- 不通过的位置会提示哪条 opcode 有问题。

失败排查原则：

1. 一次只修一条 opcode；
2. 修复后重新跑完整测试；
3. 保存失败时的 PC 和寄存器；
4. 对照指令规范，不只对照参考代码。

验收：

- [ ] 条件跳过组通过；
- [ ] 调用/返回通过；
- [ ] 算术与逻辑组通过；
- [ ] 内存传输和 BCD 通过。

## Test 4：Flags Test

测试 ROM：

```text
01_chip8_tests/bin/4-flags.ch8
```

重点检查：

- 加法进位；
- 减法借位；
- 反向减法；
- 左移/右移移出位；
- 当 `VF` 同时也是 X 或 Y 时，结果是否被错误覆盖。

推荐实现方式：先保存原值。

```c
uint8_t vx = vm->V[x];
uint8_t vy = vm->V[y];
uint8_t result = vx + vy;
uint8_t carry = ((uint16_t)vx + (uint16_t)vy) > 0xFF;

vm->V[x] = result;
vm->V[0xF] = carry;
```

但要注意：当 `x == 0xF` 时，不同测试可能要求严格的写入顺序。遇到问题应查看测试说明和选定的兼容模式。

验收：

- [ ] `8XY4` 进位正确；
- [ ] `8XY5` 非借位标志正确；
- [ ] `8XY7` 非借位标志正确；
- [ ] `8XY6/8XYE` 移出位正确；
- [ ] `VF` 特殊情况符合选择的模式。

## Test 5：Quirks Test

Quirks 是不同 CHIP-8 家族之间的历史行为差异。

常见差异：

### 1. Shift 行为

一种模式：

```text
VX = VX >> 1
```

另一种模式：

```text
VX = VY >> 1
```

### 2. `FX55/FX65` 后是否增加 I

一种模式：保持 `I` 不变。

另一种模式：操作后增加 `I`。

### 3. 绘制边界

- 裁剪；
- 环绕。

### 4. `BNNN` 行为

不同扩展可能使用 `V0` 或 `VX` 作为偏移。

第一版建议明确只支持：

```text
经典 CHIP-8 模式
```

并把选择写入项目 README。不要为了让所有模式同时通过而混合行为。

验收：

- [ ] 测试中选择的兼容模式行为一致；
- [ ] README 已记录 Shift、Load/Store、Jump、Clip/Wrap 设置。

## Test 6：Keypad Test

重点：

```text
EX9E  按键已按下则跳过
EXA1  按键未按下则跳过
FX0A  等待按键并把键值写入 VX
```

常见错误：

- SDL 键码与 CHIP-8 键值映射错误；
- 只处理 KEYDOWN，不处理 KEYUP；
- `FX0A` 每帧重复写入；
- 等待按键时 PC 仍继续执行；
- 键值越过 0～15。

推荐键盘映射：

```text
CHIP-8       PC
1 2 3 C      1 2 3 4
4 5 6 D      Q W E R
7 8 9 E      A S D F
A 0 B F      Z X C V
```

验收：

- [ ] 16 个键全部可识别；
- [ ] 按下和释放状态正确；
- [ ] `FX0A` 能暂停并恢复。

---

## Test 7：Beep Test

重点不是声音好听，而是：

- `sound_timer > 0` 时有声音；
- 归零后停止；
- 定时器按 60 Hz 下降；
- CPU 速度变化不改变蜂鸣时长。

第一版可以使用简单方波或固定音调。

验收：

- [ ] 有声与无声切换正确；
- [ ] 不会持续卡住发声；
- [ ] CPU 频率改变时声音时长基本不变。

## 四、建议的单元测试

测试 ROM 之外，还应给核心逻辑写小测试。

### 1. 字段解析

```text
6A3F → X=A，NN=3F
D125 → X=1，Y=2，N=5
1ABC → NNN=ABC
```

### 2. 加法进位

```text
V1=200，V2=100
8XY4 后：V1=44，VF=1
```

### 3. 减法借位

```text
V1=5，V2=10
8XY5 后：V1=251，VF=0
```

### 4. 调用和返回

```text
PC=0x202
执行 2300
栈保存 0x202
PC=0x300
执行 00EE
PC 恢复 0x202
```

### 5. XOR 碰撞

同一精灵在同一坐标绘制两次：

```text
第一次：像素亮，VF=0
第二次：像素灭，VF=1
```

### 6. BCD

```text
VX=231
FX33 后 memory[I..I+2] = 2,3,1
```

## 五、调试步骤

测试失败时按顺序做：

1. 固定随机数，排除随机影响；
2. 降低 CPU 速度；
3. 开启 opcode 日志；
4. 记录最后一个正确画面；
5. 找到失败测试依赖的指令；
6. 对比失败前后的寄存器；
7. 检查 `PC` 是否多跳/少跳；
8. 检查 `VF` 写入顺序；
9. 检查内存和坐标边界；
10. 修复后重新跑前面的全部测试。

不要只跑当前失败测试。后面的修改可能破坏已经通过的指令。
